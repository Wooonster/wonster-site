<h2 id="什么是-harness智能体的挽具">什么是 Harness：智能体的「挽具」</h2>
<div className="hpara">
        在生成式 AI 工程里，<strong>Harness（护具 / 挽具 / 脚手架）</strong>这个词一度被用得很乱：有人用它指代整个产品（如 Claude Code），有人用它指代评测脚本（如 SWE-bench harness）。论文《What makes a harness a harness》<a className="cit" href="#ref-1">1</a> 专门为它正本清源——要定义一个术语，先要追溯它的<em>隐喻一致性</em>。<span className="ep paper">paper</span>
</div>
<div className="hpara">
        这个词源自 12 世纪古法语 <code>harneis</code>（战争装备），14 世纪后转指「马具」：把役畜连接到车辆上的带子。它的核心隐喻非常贴切——
      </div>
<div className="diagram">
<div className="dtitle">挽具隐喻：力量 + 导向 = 有效工作</div>
<div className="flow">
<div className="fc-node a"><b>马（Model）</b><span>拥有 brute force 蛮力，但缺乏方向，可能乱跑</span></div>
<div className="fc-arrow">+</div>
<div className="fc-node g"><b>挽具（Harness）</b><span>把力量安全、高效地引导到正确方向上</span></div>
<div className="fc-arrow">=</div>
<div className="fc-node b"><b>拉动车（Task）</b><span>完成任务，且过程可控、可审计</span></div>
</div>
<div className="dcap">在 AI 语境里：模型是有蛮力但缺方向的「马」，任务是「车」，而 Harness 是确保力量被安全引导的工程层。它历经「软件测试护具 → ML 评估护具 → Agent 运行时护具」的演变，关键转变在于<strong>从「事后评估」走向「运行时介入」</strong>。<span className="ep paper">paper</span></div>
</div>
<h3 id="11-四个必要且充分条件t1t4">1.1 四个必要且充分条件（T1–T4）</h3>
<div className="hpara">
        论文给出了判定一个系统是否为 Agent Harness 的四项测试，<strong>必须全部满足</strong>才算数。这把模糊的「改文字」变成了明确的工程清单：
      </div>
<div className="stack">
<div className="layer l1"><div className="lname">T1 · Loop</div><div className="ldesc">运行时维护「推理 → 动作 → 观察」的循环。没有循环，它只是一次性的生成器。</div></div>
<div className="layer l2"><div className="lname">T2 · Tools</div><div className="ldesc">提供让模型<strong>感知并修改</strong>外部环境的工具接口（读写文件、执行命令）。只能读不能改，则不满足。</div></div>
<div className="layer l3"><div className="lname">T3 · Context</div><div className="ldesc">主动决定哪些信息进入 / 离开上下文窗口（RAG、压缩）。简单的「满溢截断」不算，必须是任务相关的筛选。</div></div>
<div className="layer l1"><div className="lname">T4 · Control</div><div className="ldesc">包含<strong>独立于模型意愿</strong>的控制机制（沙箱、Token 配额、确定性校验）。这是把 Harness 与普通 Prompt 包装层区分开的核心。</div></div>
</div>
<div className="hpara">用这把「手术刀」去切，常见系统的归类立刻清晰：</div>

<tbody>

<tr><td>代码自动补全（Autocomplete）</td><td className="down">✗</td><td className="down">✗</td><td>✗</td><td>✗</td><td>IDE Plugin（被排除）</td></tr>

</div>

<section id="s2">
<span className="sec-num">SECTION 02</span>
<h2 id="从-harness-到-auto-harness为何要自动化">从 Harness 到 Auto-Harness：为何要自动化</h2>
<h3 id="21-harness-究竟有多关键">2.1 Harness 究竟有多关键？</h3>
<div className="hpara">
        如果 Harness 只是无足轻重的「外壳」，自动化它就没意义。但多项研究给出了反直觉的实证：<strong>在模型不变的前提下，仅仅改 Harness 就能带来巨大的性能差距</strong>。
      </div>
<div className="grid2">
<div className="mini">
<h4>同一基准，6× 差距</h4>
<div className="hpara">Meta-Harness<a className="cit" href="#ref-3">3</a> 指出：在同一基准上，<strong>只改 harness</strong> 就能造成约 <strong>6×</strong> 的性能差距——这意味着 harness 的重要性不亚于模型权重本身。<span className="ep paper">paper</span></div>
</div>
<div className="mini">
<h4>78% 失败源于「非法动作」</h4>
<div className="hpara">AutoHarness<a className="cit" href="#ref-2">2</a> 的切入点：在 Kaggle GameArena 国际象棋竞赛中，<strong>78% 的 Gemini-2.5-Flash 对局失败源于非法走子</strong>，而非棋力不足。这是一个工程问题，不是模型问题。<span className="ep paper">paper</span></div>
</div>
</div>
<div className="hpara">
        于是研究者的思维框架发生转移：从「<em>如何让模型更聪明</em>」转向「<em>如何为模型设计更好的执行环境</em>」。这就催生了一个新的研究子方向——<strong>Harness 工程学（Harness Engineering）</strong>，以及它的自动化形态 <strong>Auto-Harness</strong>。
      </div>
<h3 id="22-自动化-harness-难在哪三大挑战">2.2 自动化 Harness 难在哪？三大挑战</h3>
<div className="hpara">
        AHE（Agentic Harness Engineering）<a className="cit" href="#ref-4">4</a> 一文精准概括了「为什么 harness 设计至今仍高度依赖人工、缺乏系统化方法」：<span className="ep paper">paper</span>
</div>
<div className="cols4 three-up">
<div className="col">
<div className="ct">① 异构动作空间</div>
<ul>
<li>可编辑组件性质迥异</li>
<li>提示词 / 工具实现</li>
<li>中间件 / 长期记忆</li>
<li>难在统一空间里协同优化</li>
</ul>
</div>
<div className="col">
<div className="ct">② 海量轨迹埋没信号</div>
<ul>
<li>单任务轨迹动辄数百万 token</li>
<li>关键失败根因被淹没</li>
<li>诊断信息可达 10⁷ token / 次</li>
<li>压缩反馈会丢因果链</li>
</ul>
</div>
<div className="col">
<div className="ct">③ 编辑效果难归因</div>
<ul>
<li>组件间高度耦合</li>
<li>提升 / 下降说不清是谁的功劳</li>
<li>容易陷入盲目试错</li>
<li>甚至导致性能崩溃</li>
</ul>
</div>
</div>
<h3 id="23-范式转变从参数化直觉到外置的显式推理层">2.3 范式转变：从「参数化直觉」到「外置的显式推理层」</h3>
<div className="hpara">
        Auto-Harness 的深层意义，是把智能体的部分「智能」从模型参数里释放出来，转移到<strong>可演进、可审计的代码脚手架</strong>中。这带来几个范式层面的判断：<span className="ep inf">inferred</span>
</div>
<ul className="clean">
<li><strong>外置的显式推理层</strong>：Kahneman 的「快思考 / 慢思考」框架中，LLM 的参数直觉近似于快速、模式匹配式的 System 1；而验证、约束、纠错等逻辑判断——即 System 2——被 Auto-Harness 以透明代码的形式搬到模型外部，使其可观测、可调试、可版本控制。</li>
<li><strong>大模型驱动的符号合成</strong>：传统 Neuro-Symbolic AI（神经网络 + 人工规则）的瓶颈在于规则须由人类手写，难以扩展；Auto-Harness 让 LLM 本身充当「规则编写者」，以可执行代码为介质自动生成符号约束，绕开了人工规则的扩展壁垒。</li>
<li><strong>脚手架中心化评价</strong>：未来评估的对象不再是「哪个模型更强」，而是「模型 + 环境脚手架」的综合体。《Stop Comparing LLM Agents Without Disclosing the Harness》对此有明确论证：在不披露 harness 配置的前提下对比不同 LLM agent，结论在方法论上是无效的——harness 造成的性能差距完全可能掩盖模型真实能力的差异。<span className="ep ext">external</span></li>
</ul>
<div className="note green">
<div className="tag">一句话</div>
<div className="hpara">Auto-Harness 把 LLM 从一个「容易产生幻觉的博学者」，转变为一个「能够制造精密工具的工匠」——与其强迫模型记住所有规则，不如让它<strong>写出一套能遵守规则的代码</strong>，把可靠性的保证从参数空间转移到可审计的执行层。<span className="ep inf">inferred</span></div>
</div>
</section>

<section id="s3">
<span className="sec-num">SECTION 03</span>
<h2 id="auto-harness-的实现路径">Auto-Harness 的实现路径</h2>
<div className="hpara">
        虽然各家方法名字不同，但「自动化设计 Harness」在结构上可统一为一个闭环：<strong>提议 → 评估 → 诊断 → 精炼</strong>。差异主要在三处——搜索<em>对象</em>是什么、给 proposer 看<em>什么反馈</em>、以及如何<em>保证演进不退步</em>。
      </div>
<div className="diagram">
<div className="dtitle">Auto-Harness 的通用闭环</div>
<div className="flow">
<div className="fc-node a"><b>① Propose 提议</b><span>proposer（通常是 coding agent）生成新的 harness 代码版本</span></div>
<div className="fc-arrow">→</div>
<div className="fc-node g"><b>② Evaluate 评估</b><span>在任务集上 rollout，记录分数、轨迹、错误信息</span></div>
<div className="fc-arrow">→</div>
<div className="fc-node b"><b>③ Diagnose 诊断</b><span>分析失败根因，蒸馏成可操作的证据</span></div>
<div className="fc-arrow">→</div>
<div className="fc-node a"><b>④ Refine 精炼</b><span>有针对性地改组件，必要时回滚无效改动</span></div>
</div>
<div className="fc-loop">↺ 每轮迭代写回历史库（population / archive），供下一轮诊断与提议复用</div>
</div>
<h3 id="31-基石adas--把智能体设计放进代码空间">3.1 基石：ADAS —— 把智能体设计放进「代码空间」</h3>
<div className="hpara">
        ADAS（Automated Design of Agentic Systems，ICLR 2025）<a className="cit" href="#ref-6">6</a> 奠定了思想基础：把「自动设计智能体」形式化为
        <strong>搜索空间 × 搜索算法 × 评估函数</strong>三元组，并论证<strong>代码空间是最优搜索空间</strong>——因为它图灵完备（理论全覆盖）、能复用现成开源工具、且天然契合 LLM 的编程先验。其算法 Meta Agent Search 让一个「元智能体」基于不断增长的存档（archive）阶梯式地编写越来越好的智能体代码。<span className="ep paper">paper</span>
</div>
<div className="hpara">用形式化语言表达，Auto-Harness 求解的是：<strong>在固定模型权重 $M$ 的前提下，找到能最大化所有任务实例上期望奖励的最优 Harness $H^\*$。</strong><span className="ep paper">paper</span></div>
<div className="eq">
<div className="lbl">在固定模型权重下，最大化期望任务奖励的 Harness 搜索问题</div>

$$
H^* = \arg\max_{H \in \mathcal{H}}\;
\mathbb{E}_{x \sim \mathcal{X},\; \tau \sim p^M(H, x)}
\; r(\tau, x)
$$

</div>
<div className="hpara">
        其中 $H \in \mathcal{H}$ 是搜索空间中的候选 harness 程序；$M$ 是<strong>冻结不变</strong>的基础模型；$x \sim \mathcal{X}$ 是从任务分布中采样的实例；$\tau \sim p^M(H, x)$ 是模型在 harness $H$ 下运行任务 $x$ 所产生的轨迹；$r(\tau, x)$ 是轨迹级任务奖励。被优化的唯一变量是 $H$——这正是 Auto-Harness 与参数微调路线的本质区别：无需修改模型权重，只改外部执行框架。<span className="ep paper">paper</span>
</div>
<h3 id="32-四种代表性实现">3.2 四种代表性实现</h3>
<div className="hpara">它们沿着 ADAS 的思路，各自攻克了闭环自动化中的一个核心难题：</div>
<div className="grid2">
<div className="mini">
<h4>AutoHarness<a className="cit" href="#ref-2">2</a>（Google DeepMind）</h4>
<div className="hpara"><strong>核心问题</strong>：LLM 产生非法动作的根源不在「棋力」，而在缺乏对规则的确定性执行——Kaggle 国际象棋竞赛中 78% 的 Gemini-2.5-Flash 失局源于非法走子。<span className="ep paper">paper</span></div>
<div className="hpara"><strong>方法</strong>：以 Thompson 采样引导的树搜索，迭代合成可执行代码 harness。每轮循环中，Critic 模块从环境错误信息与奖励中提炼诊断摘要，Refiner 依此生成改进的代码候选，再按采样概率选取下一个待精炼节点。定义三级接入强度：</div>
<ul className="clean">
<li><b>action-filter</b>：代码生成合法动作集，LLM 从中排序选优，保留模型推理能力</li>
<li><b>action-verifier</b>（主变体）：LLM 自由提议，代码验证合法性；非法则附带警告信息重提</li>
<li><b>action-policy</b>：代码彻底取代 LLM 决策，推理阶段零模型调用，仅执行 Python</li>
</ul>
<div className="hpara"><strong>结果</strong>：145 款游戏全部达到 <strong>100% 合法动作率</strong>；verifier 平均 14.5 次迭代收敛，Flash 小模型借此击败 GPT 大模型。<span className="ep paper">paper</span></div>
</div>
<div className="mini">
<h4>Meta-Harness<a className="cit" href="#ref-3">3</a>（Stanford / MIT）</h4>
<div className="hpara"><strong>核心问题</strong>：harness 的影响是长链路的——一次早期的记忆写入或检索策略决定，可能在数十步后才反映到最终输出。已有方法把历史压缩成分数或摘要（反馈量 10²–10⁴ tokens），而完整的 harness 诊断信息可达 <strong>10⁷ tokens</strong>，三个量级的丢失让根因追踪几乎不可能。<span className="ep paper">paper</span></div>
<div className="hpara"><strong>方法</strong>：让 coding agent（Claude Code Opus-4.6）通过 filesystem 接口，用 <code>grep</code> / <code>cat</code> / <code>diff</code> 按需查阅所有历史候选的完整源码、执行轨迹与评分——不把历史一股脑塞进 prompt，而是让 agent 主动决定看什么、怎么形成因果假设。维护 population H 中所有已评估候选，无父节点约束（proposer 可参考任意历史节点），返回准确率-上下文成本的 <strong>Pareto frontier</strong>。</div>
<div className="hpara"><strong>关键消融</strong>：scores-only 34.6 → +summary 34.9（摘要几乎无益）→ <strong>+raw traces 50.0</strong>（median），OOD 泛化 73.1% vs 基线 70.2%，且上下文成本更低。<span className="ep paper">paper</span></div>
</div>
<div className="mini">
<h4>AHE（复旦 / 北大）</h4>
<div className="hpara"><strong>核心问题</strong>：harness 的组件高度耦合——提示词、工具实现、中间件、记忆调度器等同时存在，一次编辑的效果归因极难，盲目试错甚至会导致性能崩溃（消融中仅改提示词的变体下降 −2.3 pp）。<span className="ep paper">paper</span></div>
<div className="hpara"><strong>方法</strong>：引入<strong>三层可观测性</strong>，把每次编辑变成一份可事后核查的「可验证合同」：</div>
<ol>
<li><strong>组件可观测</strong>：将 harness 拆解为 7 类解耦文件（提示词、工具实现、中间件、记忆策略、调度器、验证器、配置），每类独立版本化</li>
<li><strong>经验可观测</strong>：Agent Debugger 从原始失败轨迹中蒸馏出结构化根因摘要，避免 10⁷ token 轨迹淹没诊断信号</li>
<li><strong>决策可观测</strong>：编辑前，coding agent 须写出 <em>Change Manifest</em>——声明「我修复了什么 / 可能破坏什么」；框架在编辑后自动核对并在预期不符时触发 <strong>Git 回滚</strong></li>
</ol>
<div className="hpara"><strong>结果</strong>：Terminal-Bench 2 Pass@1 69.7%→<strong>77.0%</strong>；Change Manifest 机制有效遏制了级联式性能回退。<span className="ep paper">paper</span></div>
</div>
<div className="mini">
<h4>Self-Harness<a className="cit" href="#ref-5">5</a>（上海 AI Lab）</h4>
<div className="hpara"><strong>核心问题</strong>：AutoHarness、Meta-Harness、AHE 都隐含依赖「有更强外部模型可用于提议或评估」，但对于前沿模型而言，这个假设不成立——没有更强的同类可供参照。<span className="ep paper">paper</span></div>

<h2 id="医疗-harness-的代表系统baichuan-m4-深读">医疗 Harness 的代表系统：Baichuan-M4 深读</h2>
<div className="hpara">

<div className="hpara">它的目标可以形式化为：在给定患者历史状态 $H_t$、当前输入 $x_t$（文本 / 图像 / 工具输出）下，最大化跨多轮轨迹的期望累积临床质量奖励，同时约束幻觉率并满足安全边界。<span className="ep inf">inferred</span></div>

$$
\max\; \mathbb{E}\!\left[\sum_t r_t^{\mathrm{clinical}}\right]
\quad
\mathrm{s.t.}\;
\text{幻觉率} \le \epsilon,\;
\text{满足安全护栏}
$$

<h3 id="52-baichuan-harness作为统一运行时的四大机制">5.2 Baichuan-Harness：作为统一运行时的四大机制</h3>
<div className="hpara">
        Harness 的核心使命是消除 <strong>Sim-to-Real Gap</strong>——让 RL 训练时的环境（工具接口、行为约束、记忆存取方式）与真实部署时<strong>完全一致</strong>。它通过四个机制实现：<span className="ep paper">paper</span>
</div>
<div className="grid2">
<div className="mini">
<h4>① 异步 SubAgent 调度</h4>
<div className="hpara">主 Agent 把高上下文负荷任务（文献检索、长病历整理）分解给专用 SubAgent 并行处理，释放主 Agent 的推理资源，避免被上下文淹没。</div>
</div>
<div className="mini">
<h4>② 动态角色切换</h4>
<div className="hpara">多轮问诊中，专用「信息采集 SubAgent」完成病史收集后触发结束信号，系统切回主 Agent 做综合分析。这种「人格隔离」增强了系统稳定性。</div>
</div>
<div className="mini">
<h4>③ 多维患者记忆 + 合规写入链</h4>
<div className="hpara">短期记忆（Session Context）存瞬态信息；长期记忆（Long-term Profile）只有经过「提取 → 置信度评估 → 必要时用户确认」三步合规链后才写入，<strong>从源头过滤未核实的自述</strong>，防止长期幻觉积累。</div>
</div>
<div className="mini">
<h4>④ 渐进式隐私披露</h4>
<div className="hpara">默认只给模型看 Profile 摘要；仅当 Agent 明确发出深度回溯指令时，才按需加载底层原始电子病历。在「最小隐私暴露」与「最大计算效率」之间取得平衡。</div>
</div>
</div>

<div className="hpara">自建基准存在分布泄露风险（M4 的 Citation 90.0 即为警示）。医疗 Agent 亟需<strong>独立、动态、防泄露</strong>的交互式基准（如 AgentClinic 方向）。<span className="ep inf">inferred</span></div>
</div>
<div className="mini">

<div className="hpara">外循环 / 自动搜索一旦放开，可能引入未预见的失败模式。如何形式化「演进后系统不违反安全不变量」仍是开放问题。<span className="ep inf">inferred</span></div>

<section id="s8">
<span className="sec-num">SECTION 08</span>
<h2 id="结语从更强的模型到更好的系统">结语：从「更强的模型」到「更好的系统」</h2>
<div className="hpara">
        这篇研读的主线，是一次研究视角的位移：<strong>从「如何让模型更聪明」转向「如何为模型设计更好的运行环境」</strong>。Harness 把这层环境定义清楚（四条件），Auto-Harness 把「设计环境」本身变成可搜索、可演化的工程问题，而医疗领域则因为它极致的安全约束，把 Harness 从「外壳」逼成了「底座」。
      </div>
<div className="hpara">
        Baichuan-M4 给出了目前最完整的答案样本：用统一运行时弥合训练-部署鸿沟、用 SPAR++ 把医疗安全写进奖励、用三层闭环让系统从真实反馈中自进化。它的不足（缺消融、自建基准、难复现）同样诚实地提醒我们——<strong>医疗 Agentic AI 的工程完整性已经领先于它的科学验证</strong>。
      </div>
<div className="note">
<div className="tag">最后一句</div>
<div className="hpara">在通用领域，Auto-Harness 是效率的杠杆；在医疗领域，它更是<strong>安全与信任的基础设施</strong>。可自动化的部分尽管自动化，但「确定性安全护栏 + 强人审 + 可回滚」这三道闸门，短期内不会、也不应该交给概率。<span className="ep inf">inferred</span></div>
</div>
</section>
