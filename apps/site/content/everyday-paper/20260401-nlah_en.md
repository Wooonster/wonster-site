## Section 1 — Problem and Motivation

**What specific problem does this paper address?**
Agent harnesses — the orchestration layers governing multi-step model calls, tool delegation, artifact management, and verification — are currently implemented as runtime-specific code scattered across controllers, framework defaults, and tool adapters. The paper asks: can harness control logic be externalized as a portable, executable natural-language artifact, enabling meaningful comparison, ablation, and transfer across agent systems? Formally, the problem is defining a representation class for harness control that is simultaneously (a) executable under a shared runtime, (b) semantically equivalent to its code counterpart, and (c) modularly composable and ablatable.

**Why do existing methods fail here?**
Existing systems entangle harness logic with runtime implementation [paper]. You cannot compare two harnesses cleanly because each embeds different assumptions in its controller. Text-based prompt formats (system prompts, AGENTS.md) carry only per-call guidance, not multi-step control semantics like contracts, delegation boundaries, or durable state [paper]. Workflow compilers (AutoFlow, AgentSpec) translate natural language to intermediate representations owned by specific runtimes, losing portability [paper, inferred]. None provide a shared runtime charter that separates infrastructure semantics from harness-level control.

**Why does this problem matter?**
Harness engineering has emerged as the primary determinant of agent robustness in practice [paper cites practitioner accounts]. Making harnesses comparable artifacts would allow the field to ablate design patterns cleanly — something currently impossible when each harness is a bespoke code bundle. The paper's RQ3 result, where the NLAH reconstruction outperforms native OS-Symphony code (47.2% vs. 30.4% on OSWorld), suggests the externalization itself, not just the underlying LLM, can drive performance improvements.

## Section 2 — Technical Method

**Core contribution:** This paper proposes Natural-Language Agent Harnesses (NLAHs) — structured natural-language representations of multi-step agent control interpreted by the Intelligent Harness Runtime (IHR), enabling harness logic to be externalized, transferred, and ablated independently of controller code in a way that prior code-embedded or workflow-compiled harnesses could not support.

**Pipeline:**
- *Input encoding:* Task input enters via IHR. The runtime reads: (1) the NLAH specification (structured natural-language document defining contracts, roles, stage topology, adapters, state semantics, failure taxonomy), (2) current environment state, (3) shared runtime charter (fixed skill defining execution semantics). All state is externalized to path-addressable file artifacts (TASK.md, task_history.jsonl, artifact directories).
- *Key architectural components:*
  - **IHR**: LLM-in-the-loop runtime. At each step, LLM reads NLAH + state + charter, selects compliant action.
  - **Contracts**: Explicitly specify required inputs/outputs, budgets, permission scope, retry/stop conditions.
  - **File-backed state**: Artifacts written to named paths; later stages reopen by exact path. Survives context truncation and delegation.
  - **Multi-agent interface**: spawn_agent / wait_agent calls create child agents bounded by explicit contracts.
- *Training objective:* None — this is a runtime design, not a trained model.
- *Training/inference gap:* Not applicable.

**What's actually new:** Prior work uses natural language for per-call prompts (context engineering) or compiles it to runtime-owned intermediate representations. NLAHs make harness control — roles, contracts, verification gates, delegation boundaries, durable state semantics — a first-class natural-language object interpreted directly by an LLM runtime. The logical delta: prior work assumes harness structure must be code; this paper tests whether an LLM in the runtime loop can interpret structured natural language with sufficient fidelity to match or exceed code harnesses.

**Complexity:**
- Token overhead: NLAH-based traces are denser (58.5 logged events vs. 18.1 native steps in OS-Symphony migration) [paper], increasing total prompt tokens ~37% (11.4M → 15.7M).
- Wall-clock time: NLAH runtime is faster in OSWorld migration (140.8 min vs. 361.5 min native) [paper], primarily because child delegation topology is restructured to avoid redundant sub-searches.
- LLM calls: Dramatically reduced (34 vs. 1,200 in native) [paper] — reflecting topological reorganization more than raw efficiency.

## Section 3 — Experimental Evidence

**Main results:**

| Dataset | Metric | Prior (code baseline) | NLAH (this paper) | Δ |
|---|---|---|---|---|
| SWE-bench Verified (TRAE full IHR) | Task success | — | 74.4% | — [paper] |
| SWE-bench Verified (Live-SWE full IHR) | Task success | — | 72.8% | — [paper] |
| SWE-bench Verified (self-evolution module) | Task success | Basic 75.2% | 80.0% | +4.8 [paper] |
| OSWorld (file-backed state module) | Task success | Basic 41.7% | 47.2% | +5.5 [paper] |
| OSWorld (NLAH vs native OS-Symphony) | Task success | Native code: 30.4% | 47.2% | +16.8 [paper] |

**Ablation findings:**
Table 3 presents a module-by-module ablation. Key findings [paper]:
- **Self-evolution** is the strongest contributor on SWE-bench (+4.8 pp), consistent across both benchmarks.
- **File-backed state** helps OSWorld (+5.5 pp) more than SWE-bench (+1.6 pp) — plausibly because OSWorld tasks fail at recovery/closure more than at intent formulation.
- **Multi-candidate search** and **Verifier** are *replacers* (swap which instances are solved) rather than *expanders* (solve more overall), and can regress when their acceptance criteria diverge from the benchmark evaluator.
- **Dynamic orchestration** has near-zero effect on SWE-bench (0.0), small positive on OSWorld (+2.7 pp).
The pattern is clear: modules help when they tighten the path from intermediate behavior to evaluator acceptance; they hurt when they add layers with misaligned acceptance criteria.

**Statistical rigor:**
- No confidence intervals on any result [paper].
- Subset sizes are small (125 SWE-bench, 36 OSWorld) — both below standard evaluation scales.
- No seeds or runs reported.
- The OSWorld ablation (36 samples) is particularly fragile; a single task flip = 2.8 pp movement.

**Potential confounds:**
- The model used (GPT-5.4, reasoning effort xhigh) is extremely capable; some of the NLAH gains may reflect better prompting that an equally capable model would achieve with any structured prompt, not specifically the NLAH format.
- The RQ3 comparison (NLAH vs. native OS-Symphony) is not controlled — different execution topologies, different verification strategies, different sub-agent structures. The comparison conflates representation format with architectural redesign.
- Small OSWorld sample (36) makes all percentage differences unreliable at face value.

## Section 4 — Critical Assessment

**Methodological concern: LLM-in-the-loop interpretation introduces opacity** [inferred, moderate]
IHR's in-loop LLM interprets the NLAH specification at each step. This means harness behavior is no longer deterministic — the same NLAH may produce different execution paths with different LLM instances or temperatures. The paper acknowledges "natural language is less precise than code" but does not quantify this variance [paper].

**Methodological concern: RQ3 conflates representation and architecture** [paper-evident, critical for RQ3 claims]
The NLAH reconstruction of OS-Symphony does not merely change representation format — it relocates search topology, changes verification strategy (screen plausibility → file-backed evidence), and restructures child delegation. The 16.8 pp gain in OSWorld therefore cannot be attributed to NLAHs as a representation format alone. This is the paper's most significant claim-scope issue.

**Experimental concern: sample sizes** [paper-evident, moderate]
36 OSWorld samples is too small for statistical conclusions. With discrete task outcomes, a ±3 task swing produces ±8.3 pp movement. The ablation results (Table 3 OSWorld column) effectively have error bars that dwarf most reported deltas.

**Claim scope: "portable and comparable" requires more evidence** [inferred, moderate]
The portability claim is central to the paper's thesis, but RQ3 (the main portability test) compares only one native harness (OS-Symphony) to its NLAH reconstruction. No evidence is provided that NLAHs written for one task family transfer to another, or that ablation results replicate across different NLAH authors.

**Honest strengths:**
- The conceptual contribution is genuinely useful: naming and formalizing the harness pattern layer (contracts, roles, stage structure, file-backed state, failure taxonomy) is a meaningful contribution regardless of empirical results.
- The module ablation methodology (Table 3) is the right experimental design for the claim. The finding that modules are replacers-not-expanders is a substantive insight.
- RQ1 process metrics (Table 4 showing 90%+ of TRAE activity in child agents) provide useful mechanistic insight into what the IHR runtime actually does.

## Section 5 — Synthesis

**TL;DR:** NLAHs externalize agent harness control logic as structured natural-language documents interpreted by an LLM-in-the-loop runtime (IHR), enabling portable, ablatable harness design. Module ablations show self-evolution and file-backed state as the strongest contributors; the NLAH reconstruction of OS-Symphony outperforms native code on OSWorld by 16.8 pp. Core caveat: the OSWorld evaluation uses only 36 samples, and the migration comparison conflates representation format with architectural redesign.

**Innovation classification:** *Method advance* — the NLAH/IHR framework is a new representation class within the established agentic orchestration framework, not a paradigm shift.

**Deployment readiness:** The runtime charter and NLAH format require the deploying LLM to be strong enough to interpret complex structured natural language reliably (currently GPT-5.4 at xhigh reasoning). The portability claim is unproven across different LLM backends. Early-stage; needs larger-scale evaluation before production use.

**Open problems:**
1. How much of the OSWorld gain comes from NLAH format specifically vs. the architectural redesign of the execution topology? A controlled ablation keeping topology fixed would answer this.
2. Do NLAHs written by one practitioner execute correctly under IHR for a different practitioner's use case, or does implicit knowledge leak into the natural language in non-portable ways?
3. What is the failure mode distribution when the in-loop LLM misinterprets a contract specification? How often does this happen and how is it detected?

**Reproduction gotchas:**
- GPT-5.4 at xhigh reasoning is required; behavior with other models is untested.
- The "shared runtime charter" is a fixed skill — its exact content materially affects all results, and slight changes to it would shift all module ablation numbers.
- OSWorld requires a full desktop environment infrastructure; 36 samples does not amortize this setup cost cheaply.
- The native OS-Symphony code baseline is not publicly specified in the paper; replicating RQ3 requires reconstructing it independently.
